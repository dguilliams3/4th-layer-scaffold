The Core Idea

Not the AI model — the system around it.

Without a System

  • Chat with an AI, get some code
  • Context window fills up → AI forgets the plan
  • Session ends → all progress lost
  • No review → bugs ship as "findings"
  • One model → one set of blind spots
  • No scope document → endless scope creep

With This System

  • Structured runs with versioned scope & task logs
  • Context compacts → agent recovers in ~10 seconds
  • Session ends → resume via Run ID, everything on disk
  • Adversarial review catches real bugs before delivery
  • Multiple models cover each other's blind spots
  • SPEC versioning prevents scope creep & retry loops
The models are commodity. The value is in the orchestration infrastructure: run directories, SPEC versioning, hooks, multi-agent coordination, and compaction-proof memory. This system makes AI work durable, resumable, and reviewable.

Scope Before Code

The SPEC is written before any implementation starts. It defines what success looks like, constraints, and "Don't Retry" sections that prevent the agent from repeating failed approaches.

## Success Criteria - Parse all 1,389 records without error - Produce quality score per department ## Don't Retry - Regex-only date parsing (fails on mixed formats, use dateutil) - Single-pass NLP (misses co-occurrence)

The Recovery Loop

When context compacts or a session swaps, the agent recovers everything from disk in seconds — no re-explaining needed.

1. Read CLAUDE.md → Active Tasks table → Run ID 2. Read SPEC_v(latest).md → Scope, constraints, Don't Retry 3. Read TASK_LOG.md → Exactly where we left off 4. Read FINDINGS_REGISTRY.md → All discoveries so far Recovery time: ~10 seconds

RUN Directories: Isolated, Timestamped Workstreams

Every task gets its own directory. Agents crash — the directory survives.

Directory Structure

Each run is an isolated workstream with standardized files that any agent can read to understand the current state.

runs/CLAUDE-RUNS/ ├── ARCHIVE.md ← completed runs log │ ├── RUN-20260304-1112-feature-auth/ │ ├── SPEC_v1.md ← scope contract │ ├── SPEC_v2.md ← scope evolved │ ├── TASK_LOG.md ← progress narrative │ ├── FINDINGS_REGISTRY.md ← all discoveries │ └── subagents/ │ ├── 20260304-1130-investigation/ │ │ └── FINDINGS.md │ ├── 20260304-1145-parser-impl/ │ │ └── FINDINGS.md │ └── 20260304-1200-review/ │ └── FINDINGS.md │ └── RUN-20260304-1456-bugfix-api/ ├── SPEC_v1.md ├── TASK_LOG.md └── subagents/ ...

SPEC Versioning

SPECs are immutable. When scope changes, a new version is created. The agent reads the latest version to know what success looks like.

SPEC_v1.md → Initial architecture SPEC_v2.md → Added council approach SPEC_v3.md → Post-council refinements SPEC_v4.md → Approved final architecture Each version records: - What changed and why - New constraints discovered - "Don't Retry" additions

init-run.sh

One command creates a fully templated run directory with SPEC and TASK_LOG ready to fill in.

$ ./init-run.sh feature-auth Created: runs/CLAUDE-RUNS/ RUN-20260304-1112-feature-auth/ ├── SPEC_v1.md (template) └── TASK_LOG.md (template)

Agents communicate through files on disk — not shared conversation context.

Key Principle
Each subagent writes to its own directory. The orchestrator reads FINDINGS.md files. Never shares context directly.
Why It Works
Agents can be any model. They can crash, compact, or swap without losing work. The file system is the source of truth.
Noise Isolation
Subagent exploration generates noise. That noise stays in their directory, not the orchestrator's context window.
File-based coordination is the core insight. The orchestrator says "investigate X, write to your FINDINGS.md" — not "here's my entire conversation, continue from here." This means subagents get clean context and the orchestrator stays focused.

The Hook System: Automated Guardrails

Shell scripts that fire on agent actions — enforcing conventions without supervision.

Hook Fires On What It Does
subagent-directory-protocol PreToolUse → Agent spawn Injects the rule: "Create a timestamped directory, write FINDINGS.md there." Every subagent automatically follows the directory protocol without being told in the prompt.
force-general-purpose-agent PreToolUse → Agent spawn Rewrites read-only agent types (Explore, Plan) to "general-purpose" so they can write FINDINGS.md. Prevents agents from being unable to deliver their work.
subagent-context PreToolUse → Agent spawn Injects project-specific identity, voice, and context into every subagent. Ensures consistency without bloating each prompt.
plan-mode-reminder PreToolUse → EnterPlanMode Reminds agents of planning discipline: write the plan to a file, not just in conversation. Prevents plans from being lost to compaction.
task-log-reminder UserPromptSubmit On every human message, reminds the agent to log progress to TASK_LOG.md. Prevents progress from living only in the conversation.
Hooks are bespoke to your team's workflow. Don't copy someone else's hook repository — build hooks that enforce your conventions. The power is in the specificity: these five hooks encode exactly how we want agents to behave.

Compaction-Proof Memory

When the context window fills up, the system auto-compresses. Your work survives because it's on disk.

The Problem

  • LLM context windows are finite
  • Long sessions trigger auto-compaction
  • Compaction summarizes & drops details
  • Agent "forgets" what it was doing
  • You re-explain the same thing again

The Solution

  • CLAUDE.md → loaded every turn (Active Tasks table)
  • SPEC_vN.md → scope & constraints on disk
  • TASK_LOG.md → progress narrative on disk
  • FINDINGS_REGISTRY.md → all discoveries on disk
  • MEMORY.md → persistent cross-session notes

The Result

  • Agent reads 4 files → fully recovered
  • Survives context compaction
  • Survives session crashes
  • Survives model swaps
  • Survives multi-day work

CLAUDE.md: The Most Powerful Config Surface

CLAUDE.md is loaded into every conversation as system context. It's effectively a persistent system prompt you control. Put your Active Tasks table here and the agent always knows what it's working on.

## Active Tasks | Run ID | Description | Status | |---------------------------------|----------------------|-------------| | RUN-20260304-1112-feature-auth | Auth system overhaul | In Progress | | RUN-20260303-0900-api-refactor | REST → GraphQL | Review | → Agent reads this on EVERY turn → Knows what it's working on, even after compaction → Self-updating: agent edits this table as work progresses

Model Selection: Right Tool for the Right Job

Not "use the best model for everything" — use the right model for each task.

Haiku
Fast & cheap.
Scaffolding, boilerplate, file creation, validation checks, docstring audits. Tasks where speed matters more than depth.
Sonnet
Capable & efficient.
Implementation, code generation, standard analysis, technical reviews. The workhorse for most tasks.
Opus
Deep reasoning.
Complex parsing, architectural decisions, adversarial review, strategic analysis. When nuance and thoroughness matter.
Other
Independent perspective.
Cross-model review (GPT, Gemini, etc). Different training = different blind spots. Use for adversarial diversity.
Multi-model review isn't about finding the "best" model. It's about covering different blind spots. In practice: Opus asks strategic questions others miss. Sonnet writes proof code. External models bring genuinely different training biases. Use this to your advantage.

Reusable Orchestration Patterns

Building blocks you can compose for any project.

Scout → Build → Review

The basic pattern. A scout investigates, a builder implements, a reviewer catches bugs. Each writes FINDINGS.md in their own directory.

1. Scout (Explore agent) → Reads codebase, writes FINDINGS.md → "Here's how auth currently works" 2. Builder (general-purpose agent) → Reads scout's findings, implements → Writes code + FINDINGS.md 3. Reviewer (Opus, adversarial) → Reads builder's output → Catches bugs, writes FINDINGS.md

Analyst Council

Multiple agents with different roles examine the same data independently. No access to each other's analysis or the plan. Surfaces things no single perspective would find.

Council Members (all Opus): → Statistician: quantitative patterns → Process Engineer: operational gaps → Storyteller: narrative coherence → Skeptic: challenges all claims Key: No shared context between members. Each works independently. Orchestrator synthesizes findings.

Dependency Chain

Multi-agent team with explicit dependency ordering. Scaffold first, then parallel implementation, then capstone. Each step reads the outputs of its dependencies.

Step 1: Scaffold [Haiku] ↓ Step 2a: Parser A [Sonnet] ──┐ Step 2b: Parser B [Opus] ──┤ ↓ │ Step 3a: Analysis [Sonnet] ←─┤ Step 3b: Assessment [Sonnet] ←─┘ ↓ Step 4: Capstone [Sonnet] (reads ALL prior FINDINGS)

Multi-Model Review

Same deliverable reviewed by different model families. Each model has different training biases and blind spots. Synthesis captures what each uniquely found.

Same document → 3 reviewers: Claude Opus → Strategic questions "Are we building what the user actually wants?" GPT / Codex → Operational pragmatism "Simplify Phase 1" Claude Sonnet → Proof code "Here's SQL proving this is unnecessary for MVP"

The Human's Role: Steer, Don't Row

You make decisions. The agents do implementation, analysis, and review.

1

Define Scope

Write or approve the SPEC. Define what success looks like, constraints, and what not to retry. This is your highest-leverage activity.

2

Make Architectural Decisions

When agents surface trade-offs or ask strategic questions, you decide. The agents provide analysis; you provide judgment.

3

Review Synthesized Output

Don't review raw code line-by-line. Review the FINDINGS.md summaries, the adversarial review results, and the synthesized conclusions.

4

Evolve the System

Add new hooks when you see repeated mistakes. Update CLAUDE.md when conventions change. The system improves over time because you refine it.

What You Don't Do

The system handles these so you can focus on decisions.

✗ Write boilerplate code ✗ Manually enforce conventions ✗ Re-explain context after compaction ✗ Review every line of agent output ✗ Coordinate agent communication ✗ Track progress in your head ✗ Remember what failed last time → Hooks enforce conventions → CLAUDE.md + disk files = memory → Adversarial review = quality gate → SPEC "Don't Retry" = no loops → File protocol = coordination → TASK_LOG = progress tracking

Getting Started

Step 1: CLAUDE.md

Start here. This is your system prompt. Add your conventions, active tasks table, and workflow instructions. Every conversation will see it.

## Workflow - Clarify scope → Init run → Build → Review ## Active Tasks | Run ID | Description | Status | |--------|-------------|--------| | (none) | | | ## Conventions - Always use TypeScript strict mode - Test before committing - Log progress to TASK_LOG.md

Step 2: Run Directory

Create your first run directory with init-run.sh. Fill in the SPEC template. Now the agent has a place to work that survives everything.

$ ./runs/CLAUDE-RUNS/init-run.sh my-task # Fill in SPEC_v1.md: ## Objective What we're building and why. ## Success Criteria 1. All tests pass 2. Documentation updated ## Constraints - No new dependencies - Must work with existing auth

Step 3: Hooks

Install the hooks that match your workflow. Start with the directory protocol hook (most impactful) and add others as you need them.

// settings.local.json { "hooks": { "PreToolUse": [{ "event": "Task", "command": "python .claude/hooks/ subagent-directory-protocol.py" }], "UserPromptSubmit": [{ "command": "python .claude/hooks/ task-log-reminder.py" }] } }

Step 4: Iterate

Run your first task through the system. See what works, what's friction. Add hooks for mistakes that repeat. Evolve CLAUDE.md as conventions solidify.

Week 1: CLAUDE.md + run directories → Basic structure, manual reminders Week 2: Add directory protocol hook → Subagents auto-organize Week 3: Add adversarial review step → Catch bugs before delivery Week 4: Add task-log reminder hook → Progress always tracked Ongoing: Refine based on what actually goes wrong
